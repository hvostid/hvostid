# k6 нагрузочные тесты

Регрессионный пакет производительности для горячих эндпоинтов
платформы. Каждый сценарий проверяет:

- **p95 < 500 мс** для чтения;
- **p95 < 700 мс** для записи;
- **доля ошибок < 1 %** при указанном количестве VU.

Если порог не выполнен, `k6` завершается с ненулевым кодом, поэтому
скрипты можно встраивать в CI / релиз-пайплайны так же, как обычные
тесты.

## Подготовка

- Установите [k6](https://k6.io/docs/get-started/installation/)
  (`brew install k6`, `apt install k6` или скачайте бинарник).
- Поднимите стенд с демо-данными:

  ```bash
  docker compose up -d
  ./scripts/seed-all.sh
  ```

  Demo-сид создаёт продавцов / покупателей, которых используют
  авторизованные сценарии (пароль для всех -- `demo1234`), а также id
  объявлений / паспортов в диапазоне `1..99`, к которым обращаются
  скрипты.

## Сценарии

| Файл                                       | Эндпоинт                | Профиль нагрузки                                                           | Авторизация             |
|--------------------------------------------|-------------------------|----------------------------------------------------------------------------|-------------------------|
| [`search-listings.js`](./search-listings.js) | `GET /api/v1/listings`  | ramp `0 -> 50` (30 с), steady `50` (1 мин), ramp `50 -> 0` (30 с)          | не требуется (публичный) |
| [`create-listing.js`](./create-listing.js)   | `POST /api/v1/listings` | постоянные `10 VU` 1 мин                                                   | вход под demo SELLER    |
| [`match-score.js`](./match-score.js)         | `POST /api/v1/match/score` | постоянные `10 VU` 1 мин                                                   | вход под demo BUYER     |

Авторизованные сценарии один раз логинятся в `setup()` и переиспользуют
access-токен во всех итерациях VU, чтобы измеряемые перцентили
отражали именно тестируемый эндпоинт, а не задержку логина.

## Запуск

Отдельный сценарий:

```bash
k6 run k6/search-listings.js
k6 run k6/create-listing.js
k6 run k6/match-score.js
```

Переопределить адрес стенда (например, staging):

```bash
BASE_URL=https://staging.example k6 run k6/search-listings.js
```

Переопределить demo-учётку, если запускаете против не-demo данных:

```bash
EMAIL=loadtest-seller@example PASSWORD='...' PASSPORT_ID=12 \
  k6 run k6/create-listing.js
```

Переопределить список listing id для сценария match-score (полезно,
когда на целевом стенде нет демо-диапазона `1..99`):

```bash
LISTING_IDS='101,102,103' EMAIL=loadtest-buyer@example PASSWORD='...' \
  k6 run k6/match-score.js
```

## Тюнинг порогов

Пороги заданы в блоке `options.thresholds` каждого скрипта и помечены
тегами (`{name:listings-search}`, `{name:listing-create}`,
`{name:match-score}`), поэтому учитывают только тестируемый эндпоинт и
игнорируют разовый login-запрос из `setup()`.

При изменении количества VU или порогов обновляйте соответствующую
строку в таблице выше, чтобы документация не расходилась с тем, что
реально проверяет скрипт.
