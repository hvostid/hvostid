# Как контрибьютить в hvostid

Спасибо, что работаете над hvostid. Прочитайте этот документ целиком в
первый раз и пробегайте глазами перед каждым PR. Идея в том, что
проверки здесь совпадают с тем, на что реально смотрят ревьюеры, --
поэтому следование этим правилам ускоряет ревью.

## Настройка проекта

После клонирования один раз поставьте тулинг для хуков:

```bash
npm install                    # commitlint + husky + lint-staged на корне репозитория
npm install --prefix frontend  # eslint + prettier + lint-staged
cp .env.example .env
```

Для повседневной разработки см.:

- [`README.ru.md`](./README.ru.md#quick-start) -- запуск всего стека через Compose.
- [`README.ru.md`](./README.ru.md#разработка) -- IDE-workflow, команды Gradle и
  Vite, работа с миграциями.
- Per-service `README.ru.md` -- зоны ответственности, эндпоинты,
  переменные окружения, зависимости от других сервисов.

## Workflow

### Именование веток

Вся работа делается в ветке, названной по своей задаче:

```
feature/TXX-short-name   # новый функционал
fix/TXX-short-name       # bug fix по уже мержнутому
```

`TXX` -- id задачи (например, `T07`); `short-name` -- kebab-case
краткое описание работы. Любое изменение должно прослеживаться до
задачи -- если для бага задачи ещё нет, сначала заведите тикет и
используйте его id. Этот же id обязателен в subject коммита (см.
ниже), чтобы ветка и коммит оставались синхронизированы.

### Сообщения коммитов -- Conventional Commits

Коммиты следуют [Conventional Commits](https://www.conventionalcommits.org/)
с одним проектным дополнением: каждый subject коммита обязан содержать
ссылку на task id.

```
<type>(<scope>): TXX <subject>
```

* `<type>` -- один из `feat`, `fix`, `docs`, `style`, `refactor`,
  `test`, `chore`, `ci`.
* `<scope>` -- опциональный, в kebab-case. Обычно затронутый сервис
  или пакет: `auth`, `listing`, `gateway`, `frontend`, `docker`.
* `TXX` -- id задачи; коммит без него отвергается.
* `<subject>` -- короткая повелительная сводка без точки в конце.

Примеры:

```
feat(listing): T07 add CRUD endpoints
fix(auth): T12 reject expired refresh tokens
ci: T43 enforce conventional commits via commitlint
```

Заголовок ограничен 100 символами. Body и footer -- стандартный
Conventional Commits, опциональные.

### Локальная проверка

Сообщения коммитов валидируются локально через Husky `commit-msg`
hook, запускающий `commitlint`. При первом клонировании выполните:

```sh
npm install
npm install --prefix frontend
```

Первая команда устанавливает корневой тулинг (commitlint, husky,
lint-staged) и прописывает Husky через скрипт `prepare`. Вторая ставит
фронтенд-тулинг (eslint, prettier, lint-staged), чтобы хук `pre-commit`
мог работать на staged-файлах фронтенда.

После этого оба хука активны на каждом следующем коммите:

* `commit-msg` -- валидирует сообщение коммита по `commitlint.config.js`.
* `pre-commit` -- запускает `lint-staged` дважды. В корне репозитория
  применяет Spotless (`./gradlew spotlessApply -PspotlessFiles=...`)
  к staged Java-файлам; конфиг лежит в
  [`lint-staged.config.cjs`](./lint-staged.config.cjs). Внутри
  `frontend/` применяет `eslint --fix` и `prettier --write` к
  staged-файлам JS/JSX/CSS/HTML/JSON.

Допустимые типы коммитов и правило task-id определены в
[`commitlint.config.js`](./commitlint.config.js).

### Форматирование кода

* **Бэкенд (Java).** Spotless подключён к каждому Gradle-модулю с
  форматтером
  [palantir-java-format](https://github.com/palantir/palantir-java-format).
  `./gradlew spotlessCheck` входит в `./gradlew check` (и значит, в
  CI). Pre-commit hook автоматически применяет `spotlessApply` к
  staged Java-файлам; `./gradlew spotlessApply` чинит остальное дерево.
* **Фронтенд (JS/JSX).** ESLint и Prettier настроены в `frontend/`.
  Используйте `npm run lint` / `npm run lint:fix` и `npm run format` /
  `npm run format:check` из `frontend/`. Pre-commit hook автоматически
  чинит staged-файлы.

Общерепозиторные настройки редактора (UTF-8, LF переносы строк,
4-пробельный отступ) зафиксированы в [`.editorconfig`](./.editorconfig).

### Обход хуков

В редких случаях (опечатки в WIP-коммитах, экстренные хотфиксы) хуки
можно пропустить через `git commit --no-verify`. Не используйте это
рутинно -- ошибки форматирования и lint должны исправляться до
коммита, а не уезжать в CI.

### Ревьюеры

Ревьюеры назначаются автоматически по
[`.github/CODEOWNERS`](./.github/CODEOWNERS). Вручную добавлять
ревьюеров не нужно: GitHub берёт их из правила, совпадающего с
затронутыми файлами.

### Pull requests

PR открывается в `main`. Шаблон PR
([`.github/pull_request_template.md`](./.github/pull_request_template.md))
запрашивает связанную задачу, что изменилось, как тестировать и
чек-лист. Фронтенд-PR дополнительно требует скриншоты затронутых
экранов.

## Code style

Форматирование делается тулами (Spotless на бэкенде, ESLint + Prettier
на фронтенде) и не обсуждается -- если `./gradlew spotlessCheck` и
`npm run lint` зелёные, форматирование в порядке. Соглашения ниже -- о
том, чего не ловят тулы.

### Бэкенд (Java)

* **Используйте общие константы.** Имена HTTP-заголовков и имена ролей
  лежат в [`common`](./common/src/main/java/ru/hvostid/common).
  Используйте `SecurityHeaders.USER_ID` / `USER_ROLES` и `UserRole.*`,
  а не строковые литералы вроде `"X-User-Id"` или `"BUYER"`. Добавить
  новую общую константу предпочтительнее, чем разбрасывать литералы.
* **Комментарии объясняют *зачем*, а не *что*.** Код, которому нужен
  абзац-нарратив, обычно надо переписать. Планка для комментария: «без
  него будущий читатель удивился бы» -- скрытое ограничение, обходной
  путь конкретного бага, инвариант, не очевидный из типов.
* **Применяйте фидбэк ревью по всему проекту.** Когда ревьюер указал
  на что-то в одном файле, в этом же PR пофиксите тот же паттерн в
  файлах-собратьях. Повторяющаяся проблема -- один комментарий, а не
  пять.
* **DTO -- это records.** Request- и response-DTO -- Java records с
  Bean Validation аннотациями. Сохраняйте границу контроллера тонкой;
  бизнес-логика -- в сервисах.
* **Избегайте placeholder-значений.** Если параметр API реально не
  используется, удалите его, а не передавайте dummy-значение. Тулинг
  пометит неиспользуемый параметр метода; не глушите предупреждение,
  изобретая данные.
* **Только ASCII English.** Исходники, комментарии и сообщения коммитов
  -- ASCII English. Никаких em-тире, smart quotes, многоточий и
  не-английского текста.

### Фронтенд (JS/JSX)

* **Только функциональные компоненты** с React-хуками. Class-компоненты
  в этой кодовой базе не используются.
* **Auth context -- источник правды** о текущем пользователе. Не
  читайте токены напрямую из `localStorage` вне `AuthContext` и
  axios-клиента.
* **Маршруты защищаются** через `<ProtectedRoute>`. Новые приватные
  страницы добавляйте оборачиванием, а не своим редиректом.
* Tailwind utility-классы предпочтительнее самописного CSS. CSS
  колоцируйте только когда Tailwind не выражает нужное.

### Конфигурация

* **Переменные окружения документируются** в `application.yml`
  затронутого сервиса (с разумным default для локального запуска) и в
  per-service README. Если новая переменная нужна в Compose, добавьте
  её в [`.env.example`](./.env.example).
* **Секретам не место в репозитории.** Токены SonarQube и реджистри --
  это GitHub Actions secrets; креды БД и MinIO в `.env.example` --
  *defaults для локальной разработки*.

## Тестирование

Тесты обязательны при изменениях поведения. Для чистых рефакторов
обычно хватает существующих тестов -- но они должны проходить после
изменения.

### Что используем

| Слой       | Тулинг                                                                                                                                                              |
|------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Unit       | JUnit 5, Mockito, AssertJ                                                                                                                                           |
| Web-слой   | `@WebMvcTest`-слайсы для логики контроллеров, `MockMvc` ассерты                                                                                                     |
| Интеграция | Testcontainers PostgreSQL через [`AbstractPostgresContainerTest`](./common/src/testFixtures/java/ru/hvostid/common/testfixtures/AbstractPostgresContainerTest.java) |
| Покрытие   | JaCoCo (`<module>/build/reports/jacoco/test/jacocoTestReport.xml`)                                                                                                  |
| Нагрузка   | k6-скрипты в [`k6/`](./k6) (поиск каталога, создание объявления, оценка совместимости)                                                                              |

### Соглашения

* **Имена тест-классов зеркалят имена production-классов** с суффиксом
  `Test` (`AuthService` -> `AuthServiceTest`).
* **Используйте общую Postgres-фикстуру** для любого теста, которому
  нужна реальная БД; не поднимайте свой контейнер.
* **Не мокайте то, что вам принадлежит.** Реальные сервисы внутри
  одного модуля прогоняются end-to-end; мокаются только HTTP-границы
  (introspection-клиент, межсервисные вызовы).
* **Тесты фронтенда** ещё не подключены (отслеживается в T22). Пока
  ручная проверка фиксируется в секции «How to test» PR.

### Запуск тестов

```bash
./gradlew build                       # все модули, включая Spotless
./gradlew :auth-service:test          # один модуль
./gradlew :auth-service:test --tests AuthServiceTest
k6 run k6/search-listings.js          # нагрузочный тест против запущенного стека
```

## Процесс ревью

### Как автор

Перед запросом ревью:

* Чек-лист в шаблоне PR заполнен.
* `./gradlew build` и `npm run build` фронтенда проходят локально.
* У нового поведения есть тесты; в описании PR расписано, как
  проверить вручную.
* Ветка ребейзнута (или смержена) на актуальный `main`. Конфликты
  разрешаются на ветке, а не в merge-коммите.
* Diff -- это ровно то, что вы хотите изменить: ни случайного output
  от тула, который проект не использует, ни постороннего рефакторинга,
  замешанного в feature-PR.

Если PR крупный, оставьте короткий ревью-гайд комментарием в diff,
направляющий ревьюеров на правильный стартовый файл.

### Как ревьюер

Ревьюеры назначаются автоматически по
[CODEOWNERS](./.github/CODEOWNERS), руками просить не нужно. На что
смотреть:

* **Корректность.** Делает ли изменение то, что просит тикет? Не
  ломает ли документированное поведение в другом месте?
* **Тесты.** Покрыты ли новые ветки кода тестами и реально ли эти
  тесты проверяют изменение?
* **Границы безопасности.** Всё, что трогает auth, фильтры gateway или
  `@PreAuthorize`, заслуживает внимательного чтения.
* **Публичные контракты.** Изменения OpenAPI, переименования полей
  DTO, изменения env-переменных и миграции откатить тяжело.
* **Согласованность.** Если комментарий применим к одному месту, он
  обычно применим к нескольким -- попросите автора пофиксить
  остальные, не пересматривайте по второму кругу.

Аппрувьте, когда изменение корректно и адекватно покрыто тестами,
даже если вы написали бы иначе. Стилистические предпочтения вне этого
документа и форматтера -- это suggestions, а не блокеры.

### Мерж

PR мержатся merge-коммитом (default GitHub merge), что сохраняет
покоммитную историю. Squash-merge -- только для тривиальных fix-PR,
где покоммитная история ничего не добавляет.
